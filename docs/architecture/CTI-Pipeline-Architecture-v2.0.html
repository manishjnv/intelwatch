1. Executive Summary & Design Principles

1.1 What This System Does

The platform ingests cybersecurity news articles (avg. 3,000 words each), processes them through a multi-stage AI pipeline to extract structured intelligence (summaries, keywords, dates, IOCs, CVEs, threat actors, MITRE ATT&CK TTPs, entity relationships), enriches findings against external threat databases (VirusTotal, Shodan, AbuseIPDB), deduplicates across sources, and persists the results into a knowledge graph with automated lifecycle management. The system serves multiple customers (tenants) who can independently configure which AI models power each subtask.

1.2 Core Design Principles

PrincipleImplementation
Accuracy FirstHybrid multi-model architecture assigns the best-performing AI model to each subtask. Extraction validated via external enrichment. Confidence scoring from multiple signals.
Modular & PluggableHexagonal (Ports & Adapters) architecture. Every component — LLM provider, enrichment source, database, message broker — is behind an interface. Swap without code changes.
Multi-Tenant IsolationTenant-scoped data at every layer. Row-level security in PostgreSQL, namespace isolation in Redis, tenant-prefixed Kafka topics, encrypted API key vault per tenant.
Event-Driven & ScalableKafka-based message bus between stages. Each stage is an independent consumer group. Scale horizontally by adding consumers. No stage directly calls another.
Cost-TransparentPer-token metering at the LLM gateway. Real-time cost tracking per tenant per subtask. Budget caps with automatic model downgrade or pause.
Secure by DefaultZero-trust networking. HashiCorp Vault for secrets. AES-256 encryption at rest. TLS 1.3 in transit. SOC 2 Type II and GDPR compliant patterns.
Easy MaintenanceClean architecture with clear dependency direction. Comprehensive typing (Python type hints + Pydantic). Feature flags for gradual rollout. Structured logging + OpenTelemetry tracing.

4. Modular Code Architecture (Hexagonal / Clean Architecture)

4.1 Architecture Pattern

The system follows Hexagonal Architecture (Ports & Adapters) combined with Domain-Driven Design. This ensures that business logic (the "domain") has zero dependencies on infrastructure (databases, LLM APIs, message brokers). All external systems are accessed through ports (interfaces) with concrete adapters that can be swapped at deployment time.

Key Benefit for Maintenance: When a new LLM provider (e.g., Mistral, Llama) needs to be added, you only write a new adapter class implementing the LLMPort interface. Zero changes to pipeline logic, domain models, or other adapters. Same applies to switching from Neo4j to Amazon Neptune, or Kafka to RabbitMQ.

4.2 Dependency Direction Rule


  Outer (Infrastructure)     Middle (Application)      Inner (Domain)
  ┌─────────────────┐      ┌──────────────────┐      ┌──────────────┐
  │ FastAPI routes   │      │ Use cases         │      │ Entities     │
  │ Kafka consumers  │─────►│ (Orchestrators)   │─────►│ Value Objects│
  │ LLM adapters     │      │                   │      │ Domain Svc   │
  │ DB repositories  │      │ Input/Output DTOs │      │ Interfaces   │
  └─────────────────┘      └──────────────────┘      └──────────────┘

  Dependencies ALWAYS point inward. Domain never imports from infrastructure.

class LLMPort(ABC):
    """Port for any LLM provider."""

    @abstractmethod
    async def complete(self, request: LLMRequest) -> LLMResponse:
        """Send a prompt and receive structured output."""
        ...

    @abstractmethod
    async def get_token_count(self, text: str) -> int:
        """Count tokens for cost estimation."""
        ...

# adapters/llm/openai_adapter.py
from ports.llm_port import LLMPort
from openai import AsyncOpenAI

class OpenAIAdapter(LLMPort):
    def __init__(self, api_key: str, model: str):
        self.client = AsyncOpenAI(api_key=api_key)
        self.model = model

    async def complete(self, request: LLMRequest) -> LLMResponse:
        response = await self.client.chat.completions.create(
            model=self.model,
            messages=request.messages,
            response_format={"type": "json_object"},
            temperature=request.temperature,
        )
        return LLMResponse(
            content=response.choices[0].message.content,
            input_tokens=response.usage.prompt_tokens,
            output_tokens=response.usage.completion_tokens,
            model=self.model,
            provider="openai",
        )

Dedup Strategy (3-Layer)

Layer 1: Exact Match (Redis Bloom Filter)
  → SHA-256 hash of normalized IOC value
  → O(1) lookup, zero false negatives
  → Catches: exact duplicate IPs, hashes, domains

Layer 2: Fuzzy Match (pgvector cosine similarity)
  → Embed IOC context using lightweight model
  → Threshold: cosine > 0.92 = likely duplicate
  → Catches: same IOC described differently across articles

Layer 3: LLM Arbitration (Gemini 2.5 Pro)
  → Only invoked when Layer 2 score is 0.80–0.92 (ambiguous zone)
  → LLM compares the two IOC contexts and decides merge/keep-separate
  → Catches: sophisticated duplicates (e.g., IP mentioned as C2 vs scanner)

6.5 Stage 4: Persistence & Lifecycle Management

This stage manages the long-term intelligence stores. It operates continuously via scheduled jobs, not per-article.

IOC Lifecycle State Machine


    ┌──────┐   insert    ┌────────┐  confirm   ┌────────┐
    │ NEW  │────────────►│ ACTIVE │───────────►│ AGING  │
    └──────┘             └────────┘            └────────┘
                           │  ▲                    │
                   archive │  │ reactivate         │ ttl_expired
                           ▼  │                    ▼
                        ┌──────────┐          ┌──────────┐
                        │ ARCHIVED │◄─────────│ EXPIRED  │
                        └──────────┘  archive └──────────┘

  Confidence Decay: score(t) = initial_score * e^(-λt)
  Where λ = 0.023 (≈ 30-day half-life)
  IOC transitions to AGING when score < 0.50
  IOC transitions to EXPIRED when score < 0.20 OR past TTL

7. LLM Gateway & Model Registry

7.1 Gateway Architecture

The LLM Gateway is the single point through which ALL LLM API calls are routed. It handles model resolution, tenant API key retrieval, fallback logic, rate limiting, token counting, cost tracking, and caching. Inspired by production patterns from Portkey and LiteLLM.

# application/services/llm_gateway.py
class LLMGateway:
    def __init__(
        self,
        config_repo: TenantConfigPort,
        vault: SecretsVaultPort,
        cost_tracker: CostTrackerPort,
        cache: CachePort,
        adapters: dict[str, LLMPort],   # {"openai": OpenAIAdapter, ...}
    ):
        self.config_repo = config_repo
        self.vault = vault
        self.cost_tracker = cost_tracker
        self.cache = cache
        self.adapters = adapters

    async def execute_subtask(
        self, tenant_id: str, subtask: str, prompt: LLMRequest
    ) -> LLMResponse:
        config = await self.config_repo.get(tenant_id)
        subtask_config = config.subtask_models[subtask]

        # 1. Check budget
        remaining = await self.cost_tracker.get_remaining_budget(tenant_id)
        if remaining <= 0:
            return self._handle_budget_exceeded(config)

        # 2. Check cache (same article + subtask = same result)
        cache_key = f"{tenant_id}:{subtask}:{prompt.content_hash}"
        cached = await self.cache.get(cache_key)
        if cached:
            return cached

        # 3. Resolve model → adapter
        primary_model = subtask_config.primary_model
        adapter = self._resolve_adapter(primary_model, tenant_id, config)

        # 4. Execute with fallback chain
        try:
            response = await adapter.complete(prompt)
        except (RateLimitError, TimeoutError, ModelError) as e:
            if config.auto_fallback and subtask_config.fallback_model:
                fallback_adapter = self._resolve_adapter(
                    subtask_config.fallback_model, tenant_id, config
                )
                response = await fallback_adapter.complete(prompt)
            else:
                raise

        # 5. Track cost
        await self.cost_tracker.record(tenant_id, subtask, response.cost_usd)

        # 6. Cache response
        await self.cache.set(cache_key, response, ttl=3600)

        return response

Key Architectural Decisions Summary

Why Hexagonal Architecture? — Every external dependency (LLM provider, database, enrichment API, message broker) is behind an interface. Adding a new LLM provider means writing one adapter file. Swapping from Kafka to RabbitMQ means one adapter change. This is the foundation of easy code maintenance.

Why Event-Driven with Kafka? — Stages are fully decoupled. Stage 1 doesn't know Stage 2 exists. Each stage can be independently scaled, deployed, and debugged. If Stage 2 fails, messages stay in Kafka and are replayed when it recovers. No data loss.

Why Multi-Model Hybrid? — Research (Di Maio 2025, MDPI; Zacharis 2025, ScienceDirect) shows that multi-task prompts degrade accuracy by 5–12% compared to specialized single-task prompts. Our 3-model approach assigns the best model to each task class while bundling related subtasks to control cost.

Why Neo4j for Graph + PostgreSQL for Records? — Inspired by OpenCTI's architecture (which uses ElasticSearch for search + Redis for cache). Graph databases excel at relationship traversal ("find all IOCs connected to APT29 through any CVE"). Relational databases excel at transactional integrity, filtering, and aggregation. Using both is the pattern every major CTI platform follows.

Why External Enrichment as a Separate Stage? — This is the single most impactful competitive feature. LLMs can hallucinate IOCs. Cross-referencing against VirusTotal/Shodan/AbuseIPDB catches false positives and adds external context that the LLM never had access to. No competitor offers this as a configurable, customer-controlled layer.
