02What's New in v4.0 vs Your Existing Architecture

After deep analysis of your v2 master prompt, v3 skills system, and competitive landscape research across CrowdStrike, Recorded Future, Mandiant, ThreatConnect, OpenCTI, and others, here are the critical upgrades.

v2/v3 Architecture
  • Python FastAPI + Celery (v2) or Node/Fastify + BullMQ (v3) — inconsistent
  • Single VPS KVM2 (4 vCPU, 8GB RAM) only
  • REST API only
  • Basic Redis caching
  • Neo4j graph with manual queries
  • Single enrichment pipeline
  • No agentic capabilities
  • No ASM integration
  • No federation protocol
  • No IOA (Indicators of Activity)
v4.0 Architecture
  • Unified Node.js/Fastify + BullMQ (2x throughput, type-safe end-to-end)
  • Scale-ready: VPS start, Kubernetes graduation path
  • REST + GraphQL hybrid API (like OpenCTI + CrowdStrike)
  • 3-layer cache: Redis L1 + Browser SWR + CDN edge
  • Agentic Graph Intelligence with autonomous risk propagation
  • Dual enrichment: real-time + batch (50% cost saving)
  • Agentic threat hunting (Claude-powered autonomous hunts)
  • Attack Surface Management module integrated
  • MISP federation + TAXII 2.1 bidirectional
  • IOA + IOC dual model (behavioral + indicator intelligence)
Key Architecture Decision: Node.js Unification

Your v2 used Python/FastAPI while v3 shifted to Node.js/Fastify. v4 commits fully to Node.js + Fastify + TypeScript end-to-end. Rationale: 2x throughput vs Express, single language for full-stack, Prisma type-safety, shared Zod schemas between frontend and backend, and the v3 skill system already committed to this stack. Python NLP workloads run as isolated microservices callable via BullMQ.

Three Enrichment Patterns

Pattern 1 Batch Enrichment

50% cost reduction via Anthropic Batch API. For non-real-time enrichment of 10+ items.

System prompt + MITRE ATT&CK corpus cached with cache_control: {"type": "ephemeral"}. Minimum batch size: 10 items. Results parsed and stored async.

Pattern 2 Real-Time + Cache

Redis-first, Claude-second. Check Redis cache → if miss, call Claude with prompt caching (90% input token savings after first call).

Cache TTLs by type: hash=7d, domain=24h, ip=1h, cve=12h, url=1h, email=24h.

Pattern 3 (NEW) Agentic Hunting

Autonomous multi-step investigation. Claude generates investigation hypotheses, executes graph traversals, correlates findings, and suggests next actions.

Inspired by CrowdStrike AIDR and Recorded Future Autonomous Hunting. Human-in-the-loop approval for actions.

Reasoning Trail (Competitive Differentiator)

Every AI enrichment output is fully auditable

Unlike competitors that output a score with no explanation, ETIP's enrichment includes: score_justification (2-3 sentences citing evidence), evidence_sources (specific MITRE techniques or known campaigns), and uncertainty_factors (what would change this score). This makes every risk score defensible in SOC reviews.

08Living Graph Intelligence

The platform's most powerful differentiator. Unlike static IOC databases, ETIP's graph continuously re-evaluates ALL connected entities when new intel arrives.

How it works: Retroactive Risk Propagation

When a node's risk score changes (e.g., an IP is newly attributed to APT28), the graph engine traverses up to 3 hops of connected nodes, applying a decay factor (0.7^distance) weighted by relationship confidence. This means historical IOCs get smarter as new intelligence arrives — a domain from 6 months ago that shares infrastructure with a newly-discovered C2 server automatically gets re-scored.

Graph Query Capabilities

Entity + N-Hop Neighbors
GET /api/v1/graph/entity/:id?hops=3
Returns entity with all connected nodes up to N hops. Configurable depth slider in UI (1-5).
Shortest Path
GET /api/v1/graph/path?from=:id1&to=:id2
Find how two entities are connected. Critical for attribution analysis.
Actor/Campaign Cluster
GET /api/v1/graph/cluster/:actorId
Full 2+ hop cluster around an actor showing all infrastructure, malware, campaigns, and IOCs.
Risk Propagation
Automatic BullMQ job on every risk score change. Traverses connected nodes, applies weighted decay, updates scores. Results visible in real-time via WebSocket.

Frontend Visualization

React Flow + D3 with color-coded nodes by entity type: ThreatActor (hexagon) Campaign (rectangle) IOC (ellipse) Malware (diamond) Vulnerability (triangle). Node size scales with risk score. Edge labels show relationship type and confidence.

11Zero-Trust Security Architecture

Every request is untrusted until proven otherwise. Defense in depth at every layer.

Auth 6 Authentication Methods

Email+Password (bcrypt), Google SSO (admin), Magic Code (6-digit OTP), Customer SAML 2.0/OIDC, TOTP MFA (10 backup codes), API Keys (bcrypt hashed, scoped, revocable)

RBAC 6 Roles + Permission Strings

Roles: super_admin, admin, analyst, hunter, viewer, api_only

Permissions: module:action (e.g., ioc:read, ioc:delete)

Enforcement: rbac('role') middleware on every route

Token JWT + Refresh Token Rotation

Access token: 15-minute expiry. Refresh token: 7-day, stored in Redis, revocable. Rotation on every refresh.

Data Encryption & Validation

Integration credentials: AES-256 encrypted. All API input: Zod validation. SQL: parameterized only (Prisma). CORS: explicit origin allowlist.

Mandatory Security Checklist (Per Module)

  • No secrets in source code or git history
  • All endpoints behind authentication (except /health)
  • Rate limiting on all public-facing routes
  • Input validation via Zod on every API input
  • Parameterized queries only (Prisma) — no string interpolation
  • CORS configured restrictively (explicit origin allowlist)
  • Security headers in Nginx: HSTS, CSP, X-Frame-Options, X-Content-Type-Options
  • Dependencies audited: npm audit
  • All mutations → immutable audit log
  • Session management: revocable, viewable per user

  • POST /api/v1/ioc/search — ES full-text
  • GET /api/v1/ioc/:id/pivot — graph neighbors
  • POST /api/v1/ioc/export — CSV/JSON/STIX2
  • GET /api/v1/graph/path — shortest path

GraphQL API (NEW)
  • query { ioc(id) { ...fields, relatedActors { ...}, malware { ...} } }
  • query { graphTraversal(from, hops, types) { nodes, edges } }
  • query { correlation(entityIds) { matches, confidence } }
  • subscription { iocCreated(severity: HIGH) { ...fields } }
  • Complex nested queries in single request
  • Real-time subscriptions via WebSocket
